1. ROLE-BASED ACCESS CONTROL (RBAC)
==================================================

1.1 Role Hierarchy
---

The system implements three distinct user roles:

  PATIENT
  - Standard user seeking healthcare services
  - Can view own appointment bookings
  - Can upload medical records
  - Can receive notifications
  - Cannot access admin features
  - Cannot manage other users
  - Authorization Level: Low

  DOCTOR
  - Healthcare professional
  - Can view own profile and schedule
  - Can accept/reject appointments
  - Can issue prescriptions
  - Cannot access admin features
  - Can view patient records (assigned only)
  - Authorization Level: Medium

  ADMIN
  - Platform administrator
  - Full access to user management
  - Can verify/approve doctors
  - Can monitor platform activities
  - Can view all user data
  - Can manage system configuration
  - Authorization Level: High


1.2 Implementation Details
---

Location: backend/services/auth/src/auth/middleware/authMiddleware.js

Role Assignment (Registration):
  1. User submits registration with email, password, name, role
  2. System validates role is one of: ["patient", "doctor", "admin"]
  3. If no role specified, defaults to "patient"
  4. Role stored in MongoDB User document
  5. Role included in JWT token payload

Token Generation:
  1. After successful login/registration, JWT token created
  2. Token payload includes: { id: userID, role: userRole }
  3. Token signed with JWT_SECRET from environment
  4. Token expires after JWT_EXPIRE duration (7 days)
  5. Client stores token in localStorage

Access Control Middleware:
  
  protect middleware:
    - Validates Bearer token in Authorization header
    - Decodes JWT token and verifies signature
    - Fetches user from database using token ID
    - Attaches user object to request (req.user)
    - Returns 401 Unauthorized if token invalid/missing

  authorizeRoles middleware:
    - Called after protect middleware on protected routes
    - Checks if req.user.role matches allowed roles
    - Returns 403 Forbidden if user role not authorized
    - Example: authorizeRoles('admin') on user listing


1.3 Protected Routes
---

Auth Service Routes:

  Public Routes (No Authentication):
    POST /auth/register           - Register new user
    POST /auth/login              - User login

  Protected Routes (Authentication Required):
    GET /auth/profile             - Get own profile (any authenticated user)
    
  Admin-Only Routes (Authentication + Admin Role):
    GET /auth/                    - List all users (admin only)


2. AUTHENTICATION MECHANISMS
==================================================

2.1 Password Security
---

Implementation: bcryptjs library (npm: bcryptjs)

Process:
  1. User submits plain-text password during registration
  2. Password validated (must be provided, not empty)
  3. bcryptjs.hash() called with:
     - plaintext password
     - salt rounds: 10 (high security)
  4. Hash generated asynchronously
  5. Hash stored in MongoDB (never plain-text password)
  6. Plain-text password discarded from memory

Hash Generation:
  Cost: 10 rounds of bcrypt hashing (2^10 = 1024 iterations)
  Time: ~1 second per hash (intentional slowdown)
  Purpose: Prevents rainbow table attacks
  Salt: Unique for each password

Login Verification:
  1. User submits email and password
  2. Database fetches user by email
  3. bcryptjs.compare() called:
     - Takes submitted password
     - Takes stored hash
     - Compares securely (timing-attack resistant)
  4. If match: authentication successful
  5. If no match: authentication fails (401 error)
  6. Never reveal if email exists or password wrong (for security)


2.2 JWT Token Authentication
---

Implementation: jsonwebtoken library (npm: jsonwebtoken)

Token Structure:
  Header:   { alg: "HS256", typ: "JWT" }
  Payload:  {
    id: "<user_mongodb_id>",
    role: "<user_role>",
    iat: <issued_at_timestamp>,
    exp: <expiration_timestamp>
  }
  Signature: HMACSHA256(header.payload, JWT_SECRET)

Token Generation (after login/register):
  jwt.sign({
    id: user._id,
    role: user.role
  }, process.env.JWT_SECRET, {
    expiresIn: process.env.JWT_EXPIRE || '7d'
  })

Secret Key:
  - Environment variable: JWT_SECRET
  - Default (dev): 'medsync_super_secret_jwt_key_2026'
  - Should be changed in production
  - At least 32 characters recommended
  - Never committed to version control

Token Expiration:
  - Default: 7 days
  - After expiration, token becomes invalid
  - Client must login again to get new token
  - Reduces window for compromised tokens

Token Verification:
  - Client sends: Authorization: Bearer <token>
  - Server extracts token from header
  - jwt.verify() validates:
    - Signature is correct
    - Token not expired
    - Token not tampered with
  - If invalid: returns 401 Unauthorized
  - If valid: decoded payload passed to next middleware


2.3 Authorization Checks
---

Protect Middleware Flow:
  1. Check Authorization header exists
  2. Extract Bearer token from header
  3. Verify JWT signature and expiration
  4. Decode token to get user ID
  5. Query database to fetch user by ID
  6. Attach user object to request
  7. Proceed to next middleware/route handler

AuthorizeRoles Middleware Flow:
  1. Check if authenticated (req.user exists)
  2. Check if user.role is in allowed roles array
  3. If not authorized: return 403 Forbidden
  4. If authorized: proceed to route handler

Example Protected Route:
  router.get('/', protect, authorizeRoles('admin'), listUsers);
  
  - protect: User must be authenticated
  - authorizeRoles('admin'): User must have admin role
  - listUsers: Only executed if both conditions met


2.4 Session Management
---

Frontend Session Handling:
  - Token stored in browser localStorage
  - Token included with every request (Axios interceptor)
  - On logout: token removed from localStorage
  - On token expiration: user redirected to login
  - No server-side session storage (stateless JWT)

Security Implications:
  - Stateless architecture: no session database needed
  - Scalable: works across multiple servers
  - XSS Risk: localStorage can be accessed by JavaScript
  - CSRF Protected: JWT not stored in cookies (used Bearer header)

3. NOTIFICATION SERVICE SECURITY
==================================================

3.1 Email Notification Security
---

Implementation: nodemailer library

Credentials Management:
  Email User: process.env.EMAIL_USER
  Email Password: process.env.EMAIL_PASS
  Stored in environment variables (not in code)
  Loaded from .env file at startup
  .env file added to .gitignore (never committed)

Gmail Setup (for development):
  1. Enable 2-Factor Authentication
  2. Generate App-Specific Password (16-character password)
  3. Use app password in EMAIL_PASS (not actual Gmail password)
  4. Critical: Use app password, not real password

Email Sending Process:
  1. Client calls POST /api/notify/email
  2. Validates email address format
  3. Validates subject and body present
  4. Calls nodemailer.sendMail()
  5. Sent asynchronously (doesn't block request)
  6. Returns success/error to client

Validation:
  - 'to' field required (destination email)
  - 'subject' field required
  - 'text' field required (email body)
  - Email format validated with basic regex

Error Handling:
  - SMTP connection failures caught
  - Invalid email address errors returned
  - Authentication failures logged
  - Never reveals credentials in error messages


3.2 SMS Notification Security
---

Implementation: Twilio library

Credentials Management:
  TWILIO_SID: Account SID
  TWILIO_AUTH_TOKEN: Authentication token
  TWILIO_FROM: Twilio phone number
  All stored in .env file as environment variables

Twilio Credentials:
  - SID: Uniquely identifies Twilio account
  - Auth Token: Like password for API authentication
  - From Number: Phone number SMS appears from
  - Store safely: never log or expose

SMS Sending Process:
  1. Client calls POST /api/notify/sms
  2. Validates phone number present
  3. Validates message content present
  4. If Twilio credentials available:
     - Initializes Twilio client with SID and token
     - Calls twilio.messages.create()
     - Specifies: to, from, body
  5. If Twilio not configured:
     - Falls back to console logging (dev only)
     - Logs to stdout for testing

Validation:
  - 'to' field required (destination phone)
  - 'message' field required (SMS body)
  - Phone number format not strictly validated
     (Twilio handles validation)

Error Handling:
  - Invalid phone numbers rejected by Twilio
  - Invalid credentials caught at initialization
  - Network errors handled gracefully
  - Returns error message to client

Fallback Mechanism:
  - If Twilio not configured in environment
  - SMS request logs to console instead of sending
  - Allows development without Twilio account
  - Production must have valid Twilio credentials


3.3 Notification API Security
---

Rate Limiting: Not implemented (recommended for production)
  - Currently: no rate limiting
  - Production: implement limits per IP/user
  - Standard: 100 requests per hour per IP

CORS: Enabled
  - Frontend can call notification endpoints
  - Backend-to-backend communication allowed

Input Validation:
  - Email address required and validated
  - Phone number required (format flexible)
  - Subject/message content required
  - Error messages returned for invalid input

No Authentication on Notification API:
  - Notification endpoints are public
  - Allows internal services to send notifications
  - Production: add authentication/credentials
  - Production: implement API key or JWT validation

4. DATABASE SECURITY
==================================================

4.1 MongoDB Connection
---

Connection String (MONGO_URI):
  Format: mongodb://[user]:[password]@host:port/database
  Default (local): mongodb://localhost:27017/medsync
  Docker: mongodb://mongo:27017/medsync (with auth)
  
Production Connection:
  - Use username and password in connection string
  - Use MongoDB Atlas (cloud) with encrypted connection
  - Enable IP whitelist in MongoDB
  - Use TLS/SSL encryption

Authentication:
  - If MongoDB has authentication enabled:
    mongodb://admin:password@mongo:27017/medsync?authSource=admin
  - Credentials stored in .env
  - Never hardcoded in source

Connection Options:
  - SSL: true (enable in production)
  - retryWrites: true (automatic retry on failure)
  - serverSelectionTimeoutMS: 5000 (timeout setting)
  - minPoolSize: 10 (connection pooling for performance)


4.2 Data Protection
---

User Document Structure:
  {
    _id: ObjectId (unique identifier),
    name: String,
    email: String (unique),
    password: String (bcrypt hashed),
    role: String (patient|doctor|admin),
    verified: Boolean (for doctors),
    createdAt: Date,
    updatedAt: Date
  }

Sensitive Data:
  - Password: Hashed with bcryptjs, never stored plain
  - Email: Unique, used for authentication
  - Role: Used for authorization decisions
  - Verified: Tracks doctor verification status

Query Security:
  - Mongoose ODM (Object Data Modeling)
  - Prepared statements (Mongoose.select)
  - No raw string concatenation in queries
  - Input validation before database operations

Password Retrieval:
  - Passwords never returned in API responses
  - User queries exclude password by default: .select('-password')
  - Only used for login comparison with bcryptjs.compare()


4.3 Data Validation
---

Input Validation (before database):

  Registration (/auth/register):
    - name: required, string, 1-100 characters
    - email: required, valid email format, unique check
    - password: required, string
    - role: optional, must be valid role

  Login (/auth/login):
    - email: required, valid format
    - password: required

Mongoose Schema Validation:
  - Field types defined in schema
  - Required fields enforced
  - Invalid types rejected before database operation
  - Custom validators for complex rules

Database Constraints:
  - Unique index on email (prevents duplicates)
  - Created indexes for query performance
  - TTL indexes not used for user data

5. API SECURITY BEST PRACTICES
==================================================

5.1 HTTP Headers
---

Implemented Headers:
  - Content-Type: application/json
  - Authorization: Bearer <token>

Recommended (not yet implemented):
  - X-Frame-Options: DENY (prevents clickjacking)
  - X-Content-Type-Options: nosniff (MIME type sniffing)
  - Strict-Transport-Security: max-age=31536000 (HSTS)
  - Content-Security-Policy: (CSP rules)

Note: These should be added in production with helmet.js


5.2 HTTP Status Codes
---

Success Responses:
  200 OK             - Successful request
  201 Created        - Resource created (registration/login)

Error Responses:
  400 Bad Request    - Invalid input (missing fields)
  401 Unauthorized   - Authentication failed or missing
  403 Forbidden      - Authenticated but not authorized for role
  404 Not Found      - Resource doesn't exist
  409 Conflict       - Email already exists (duplicate)
  500 Server Error   - Unexpected error

Security Implication:
  - 401 vs 403: Helps attackers understand access (acceptable tradeoff)
  - Never leak implementation details in error messages
  - Never reveal database structure in errors


5.3 CORS Configuration
---

Current Configuration:
  - CORS enabled (app.use(cors()))
  - Allows all origins in development
  - Allows all HTTP methods

Production Recommendation:
  app.use(cors({
    origin: ['https://medsync.com', 'https://app.medsync.com'],
    methods: ['GET', 'POST', 'PUT', 'DELETE'],
    allowedHeaders: ['Content-Type', 'Authorization'],
    credentials: true
  }))

Security Impact:
  - Prevents unauthorized cross-origin requests
  - Restricts API access to known domains
  - Protects against CSRF attacks (partial)


5.4 Input Sanitization
---

Current Approach:
  - Basic validation (required fields, type checking)
  - Email format validation
  - No special HTML sanitization

Recommended Improvements:
  - Use express-validator for comprehensive validation
  - Sanitize input strings (remove HTML/script)
  - Validate array lengths and object depths
  - Use trim() on strings to remove whitespace

Security Risk (Low):
  - User input stored in database
  - If not properly sanitized, could cause issues
  - Currently: low risk due to MongoDB, not SQL injection

6. FRONTEND SECURITY (ADMIN DASHBOARD)
==================================================

6.1 Token Management
---

Token Storage:
  Location: browser localStorage
  Key: 'token'
  Format: JWT string
  Access: localStorage.getItem('token')

Security Considerations:
  localStorage Vulnerabilities:
    - Accessible by JavaScript (XSS attacks)
    - Persists across browser sessions
    - Not automatically sent with requests (need manual header)
  
  Advantages:
    - Survives page refresh
    - Available across tabs
    - Doesn't need cookies (CSRF protection)

XSS Protection (Recommended):
  - Use HttpOnly cookies instead of localStorage
  - Content Security Policy (CSP)
  - Sanitize user-generated content
  - Validate output in React components


6.2 Authentication Flow
---

Login Process:
  1. User enters email/password on login page
  2. Form submitted to POST /auth/login
  3. Axios makes HTTP POST request
  4. Response received with token
  5. Token stored: localStorage.setItem('token', token)
  6. User redirected to dashboard

Request Headers:
  Axios interceptor (recommended to add):
    const token = localStorage.getItem('token');
    axiosInstance.defaults.headers.common['Authorization'] = 
      `Bearer ${token}`;

Token Cleanup (Logout):
  1. User clicks logout button
  2. localStorage.removeItem('token') called
  3. User state reset
  4. Redirect to login page

Session Expiration:
  - Token expires in 7 days based on JWT exp claim
  - Frontend doesn't actively check expiration
  - API returns 401 when token expired
  - Recommend: check token expiration client-side
  - Show warning before expiration (e.g., 1 hour before)


6.3 Admin Dashboard Authorization
---

Role-Based UI Rendering:
  - Admin Dashboard only visible to admin role
  - Check user.role === 'admin'
  - Redirect to login if not admin

API Calls with Authorization:
  1. Axios request includes Authorization header
  2. Server validates token and role
  3. If unauthorized: API returns 403 Forbidden
  4. Frontend handles error (show message)

Security Note:
  - Frontend authorization is UI-level only
  - Backend MUST validate authorization
  - Frontend authorization can be bypassed
  - Always validate on backend (critical)


6.4 Error Handling
---

Error Display:
  - Network errors shown to user
  - API errors displayed with messages
  - No sensitive information leaked

Recommended Improvements:
  - Log errors to server for monitoring
  - Implement error boundary components
  - Track failed login attempts
  - Alert on suspicious activity

7.2 Docker Security
---

Risks:
  - Container runs with elevated privileges (don't)
  - Environment variables visible in process list
  - Images may contain vulnerabilities
  - No resource limits (CPU, memory)

Best Practices:
  - Run container as non-root user
  - Set resource limits (memory, CPU)
  - Use minimal base images (alpine Linux)
  - Scan images for vulnerabilities
  - Implement health checks
  - Use read-only filesystems where possible

Dockerfile Recommendations:
  FROM node:18-alpine
  RUN addgroup -g 1001 -S nodejs
  RUN adduser -S nodejs -u 1001
  WORKDIR /app
  COPY package*.json ./
  RUN npm ci --only=production
  COPY --chown=nodejs:nodejs . .
  USER nodejs
  EXPOSE 5000
  HEALTHCHECK --interval=30s CMD node healthcheck.js
  CMD ["node", "server.js"]

8. THREAT PREVENTION
==================================================

8.1 Common Attacks Mitigated
---

SQL Injection:
  Status: ✓ PROTECTED
  Reason: MongoDB (NoSQL) not SQL
  Note: NoSQL injection possible but limited
  Implementation: Input validation, Mongoose ODM

XSS (Cross-Site Scripting):
  Status: ✓ PARTIALLY PROTECTED
  Reason: React escapes output by default
  Remaining: Unsanitized user input in HTML
  Recommendation: Use DOMPurify for user content

CSRF (Cross-Site Request Forgery):
  Status: ✓ PROTECTED
  Reason: JWT in Authorization header (not cookie)
  Cookie-based: Would need CSRF tokens

Brute Force (Password Guessing):
  Status: ✗ NOT PROTECTED
  Recommendation: Implement rate limiting per IP
  Better: Add account lockout after N failed attempts
  CAPTCHA: Add for repeated failures

Man-in-the-Middle (MITM):
  Status: ✓ PARTIALLY PROTECTED
  Reason: Need HTTPS in production
  Recommendation: Use TLS/SSL certificates
  HSTS: Add Strict-Transport-Security header

Token Hijacking:
  Status: ⚠️ RISK if XSS occurs
  Reason: localStorage accessible by JavaScript
  Recommendation: Use HttpOnly cookies instead

Authorization Bypass:
  Status: ✓ PROTECTED
  Reason: Backend validates role on every request
  Note: Frontend authorization bypassing doesn't work
  Principle: Never trust frontend auth


8.2 Security Monitoring
---

Recommended Implementation:
  - Log all authentication attempts (success/failure)
  - Alert on multiple failed login attempts
  - Monitor unusual access patterns
  - Track API request rates per user
  - Log database queries (sensitive operations)
  - Alert on database access outside normal times

Implementation:
  const logger = require('./logger');
  logger.info(`Login attempt by ${email}`);
  logger.error(`Failed login for ${email}`);
  logger.warn(`High request rate from ${ip}`);


8.3 Incident Response
---

If Credentials Compromised:
  1. Immediately rotate all secrets
  2. Force password reset for all users
  3. Invalidate all existing JWT tokens
  4. Review access logs for unauthorized access
  5. Notify affected users
  6. Update passwords for service accounts
  7. Audit database for suspicious changes

If Token Leaked:
  1. Reduce JWT_EXPIRE duration
  2. Implement token revocation list (blacklist)
  3. Force re-authentication for critical operations
  4. Monitor for unauthorized access using token
  5. Consider implementing token rotation
